- **Actual Output:** - Determinism
  - ML systems: deterministic — same input yields the same, exact output (e.g., calculator: 2+3 always =5). Evaluation is simple assertion-based testing.
  - LLM apps: probabilistic/non-deterministic — the same input can produce different but valid outputs (different wording, length, structure). Outputs are semantically flexible rather than exact-string matches.

- Dynamic cost
  - ML systems: cost is predictable/fixed for a given workload (example: 100 rows → cost 10 rupees).
  - LLM apps: cost is dynamic and hard to fix because both input length and output length vary; evaluation and reporting on text outputs add further cost.

- Input–output tracking & evaluation
  - ML systems: input→output relationship is stable and easily tracked; evaluation is cheap and automated using mathematical formulas and test cases.
  - LLM apps: input→output relationship is dynamic and must be actively managed; simple assertions fail because outputs vary semantically. Evaluation is costly, requires human judgment/detailed reports (human annotation doesn’t scale), and must account for multiple components (e.g., retriever vs generator) that can fail independently.
- **Expected Output:** ML systems are generally deterministic: the same input row gives the same output every time, with fixed compute/cost for that input size.  

LLM apps are probabilistic and dynamic: the same input can produce different outputs, and both cost and compute vary because input length, output length, and retrieved context all change.  

Also, ML prediction usually has a simple input-to-output flow, while LLM apps often require tracking the full journey from input through retrieval, prompting, and output.

## ✅ PASS - test_case_12

<details><summary><b>View Test Case Data</b></summary>

- **Input:** How do LLM evals differ from software tests in fixed vs. flexible output and multi-metric quality?
- **Actual Output:** - Fixed vs. flexible output
  - Traditional software: fixed, deterministic outputs. Given the same input (e.g., 2+3) the output can only be one value (5). Tests are simple assertions: pass or fail.
  - LLMs: flexible, probabilistic, non‑deterministic outputs. The same prompt can produce many different valid answers (different words, lengths, styles). Outputs are semantically flexible (not exact strings), so equality-based assertions often fail even when answers are correct.

- Multi‑metric quality vs. single correctness
  - Traditional software testing focuses on one dimension: correctness (does the program produce the expected value?).
  - LLM evaluation is multi‑dimensional. You must judge correctness, relevance, completeness, groundedness (is the answer based on the source in RAG setups vs. hallucinated), and other dimensions (clarity, faithfulness, etc.). There is a grey area — outputs can be different yet still acceptable — so evaluation requires judgment and nuanced scoring rather than binary pass/fail.

- Practical consequences
  - LLM evaluations are costlier and require detailed reports and human judgment; human annotation doesn’t scale, pushing toward automated metrics (which are themselves hard to design).
  - Multiple components (e.g., retriever and generator) can fail independently, so evaluation must consider component‑level behavior as well as end‑to‑end output.
  - Non‑determinism makes simple assertion‑based testing impossible; you need probabilistic/aggregation approaches or richer automated metrics.
- **Expected Output:** LLM evals are harder than software tests because:

- **Software tests** usually check **fixed, deterministic output** — one input should produce one expected answer, so it’s mostly **pass/fail**.
- **LLM evals** deal with **flexible, probabilistic output** — the same prompt can produce different but still valid answers.

Also, software testing mostly checks **correctness only**, while LLM evals need **multi-metric quality checks** like:
- correctness
- relevance
- completeness
- groundedness
- safety

So, unlike software tests, LLM evals are not just about “right or wrong”; they evaluate whether the answer is good across multiple dimensions.
